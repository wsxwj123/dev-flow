# PLAN — task-flow 实施方案（纯指令文件型 skill）

> 需求见 `.devflow/BRIEF.md`，架构定论见 `task-flow-设计/定论-v3架构.md`。
> task-flow 是无代码的纯指令 skill，安装到 `~/.claude/skills/task-flow/`。**完全自包含**：单独下载 `task-flow/` 就能用，不引用 dev-flow 任何文件，安全防线与跨平台映射自带一份（`references/rules.md`、`references/PLATFORMS.md`，与 dev-flow 同源、文件头互相提醒同步）。

---

## 1. 架构与设计决策

### 1.1 核心形状（决定文件怎么拆）

架构不是 dev-flow 的七阶段线性流水线，而是：**一个轻分诊入口 + 一套极简主循环 + 两道客观质量闸 + 一个随时可调的参与度旋钮 + 几个按任务类型触发的可选动作。** 文件拆分贴这个形状，不照抄 dev-flow 的按阶段（01→07）切法。

具体差别：dev-flow 的 references 是"一个阶段一个文件、顺序走"；task-flow 的 references 是"一个**机制**一个文件、按分诊结果条件触发"——立项后不一定线性经过每个文件，而是分诊结论决定这次读哪几个。

### 1.2 加载策略（照搬 dev-flow 的"进阶段前必读"机制）

- **SKILL.md 每次触发都加载**，必须精炼：只放红线级铁律、分诊三问与任务分级、主循环骨架图、参与度旋钮触发词、产物约定表、references 路由表、反模式清单。**所有细节下沉 references**，把 SKILL.md 的行数控制在 dev-flow SKILL.md 量级（~100 行），防止关键铁律被稀释。
- **references 按需加载**：进入某个机制（立项 / 过闸 / 螺旋探索 / 异常）前，先完整读对应 reference 再动手——这是对抗长会话遗忘的机制，和 dev-flow 铁律 6 一致。
- **逐字 prompt 就近内联**在会派该子代理的 reference 文件里（补遗代理开场在 01、盲评+终判开场与白名单禁令在 02、红队开场在 03），标"逐字复制、不许凭记忆改写"。《数据与指令隔离声明》逐字段落放本技能 `references/rules.md`，各处引用它。不单开 prompts.md——task-flow 派的子代理少，就近内联更不易漂移，也少一层文件跳转。

### 1.3 完全自包含（不引用 dev-flow 任何文件）

**结构性决定**：task-flow 必须自包含，单独下载 `task-flow/` 文件夹就能用，**不跨技能引用 dev-flow 的任何文件**。安全防线、隔离声明、跨平台映射全部自带一份，放 `references/rules.md` 和 `references/PLATFORMS.md`。

- `references/rules.md`：**为非开发任务定制的精简规则**（不是照搬 dev-flow 那份开发向 rules.md）——说人话铁律（含禁比喻）、《数据与指令隔离声明》逐字段落（子代理 prompt 要逐字引用它，必须原样保留）、适用的危险操作红线（不删用户文件、原始数据只读、敏感数据不进子代理/不落盘/不进 git、不暴露密钥）、Git 规范（一个逻辑变更一个 commit）、精简原则（不过度设计）。开发专属条款（代码风格、测试质量等）不搬。
- `references/PLATFORMS.md`：只自带 task-flow 用得到的跨平台对应——子代理怎么派、模型名映射（高推理/高性价比→各平台具体模型）、计划确认工具映射。
- 两个文件的**文件头都标注**："与 dev-flow 的同名文件同源，那边对应文件改进时，检查本文件要不要同步。"——用文件头互相提醒代替硬引用，避免同源内容各自漂移。

## 2. 文件结构

```
task-flow/
├── SKILL.md                    每次触发加载的主文件：红线铁律 + 任务分级与三问分诊 +
│                               主循环骨架图 + 参与度旋钮触发词 + 产物约定表 +
│                               references 路由表（何时读哪个）+ 反模式清单
└── references/
    ├── 01-立项分诊.md          进主循环前必读：grilling 拷问 → opus 补遗子代理 →
    │                           三问分诊 + 数据敏感度一问 → 让位判定（含问"要不要把关"）→
    │                           起草出厂检查单（三层来源）→ 拆解与派工计划（子任务、依赖、
    │                           派谁、各走哪条质检；主会话拟，不派盲审代理）→ 成本预估 →
    │                           计划与合同同一次给用户确认后锁定。
    │                           〔颗粒度规则（用户定）：计划只拆到"工作包"级——
    │                           专门技能承接的段落在计划里就是一行，禁止把该技能的内部
    │                           阶段抄进/复述进计划；工作包内部怎么干以技能自身拆解为准，
    │                           task-flow 只约束边界（输入/交付物/须过的合同条款）；
    │                           边界冲突以用户签的 CHECKLIST 为准打回，内部步骤永远听技能的；
    │                           通用子代理的工作包以"一个子任务=一次可交付的工作"为限，
    │                           不拆到操作步骤〕
    ├── 02-质量闸.md            过闸 / 交付前必读：契约闸门（硬闸零容忍 / 软闸给扣分位）+
    │                           证据账本（孤儿清单）+ 硬验证（分轻重）+ 软评审降级 +
    │                           查缺补漏式外包闸（只补三样）+ 抽账终判〔含逐字 prompt〕
    ├── 03-触发动作.md          按分诊结果进对应动作前读：螺旋探索（目标不清，含 N 圈无进展停）
    │                           + 红队对抗（评估型，含回合上限、逐字 prompt）
    │                           + 样品先行（长文/大分析强制，含阈值）
    ├── 04-异常与续跑.md        遇异常/续跑时读：不可行止损 + 变更单 + 用户执行节点 park +
    │                           返修入口 + 续跑留档 + 残局清理
    ├── rules.md                自带的非开发向精简规则：说人话+禁比喻铁律、《数据与指令隔离
    │                           声明》逐字段落、适用红线（不删用户文件/原始数据只读/敏感数据
    │                           不进子代理不落盘不进git/不暴露密钥）、Git 规范、精简原则。
    │                           文件头标注"与 dev-flow 同源，那边改进时检查本文件要不要同步"
    └── PLATFORMS.md            自带的跨平台对应（只留 task-flow 用得到的）：子代理怎么派、
                                模型名映射、计划确认工具映射。文件头同样标注同源提醒
```

完全自包含：单独下载 `task-flow/` 就能用，不引用 dev-flow 任何文件（见 §1.3）。

拆 4 个流程 reference + 2 个自带规则 reference：立项分诊是唯一入口聚一处；两道闸是质检核心聚一处；按类型触发的三个动作（螺旋/红队/样品）都是"条件触发、进前才读"，聚一处、读匹配当前任务类型的那一节即可；异常与续跑是另一维度（出错/接续）聚一处；rules.md / PLATFORMS.md 是全程可引的横切规则，各自成文。再拆会碎、合并会让单文件过长稀释重点。

## 3. 对外行为约定

完整版单独成文 `.devflow/INTERFACE.md`（测试设计的唯一输入）。这里给索引，六个子契约：

1. **触发契约**：非开发实质任务触发；开发→dev-flow、微任务→直接做、单技能全覆盖→让位并问一句要不要把关。
2. **六类任务预期行为序列**：对应 BRIEF 六个盲测场景（纯论文/数据分析/目标不清/混合接缝/微任务/旋钮切换），逐个写清会问什么、派什么子代理、落什么盘、在哪停；另含"未命中六类的通用撰写/总结"**默认序列**（立项→契约闸门+证据账本，不附加硬验证/红队）。
3. **产物文件契约**：`.taskflow/<任务>/` 下 BRIEF/PROJECT/CHECKLIST/EVIDENCE/VERIFY/MAP/CHANGE，含目录命名规则与同名冲突处理、唯一写者写盘协议（子代理不写盘、只回传），谁写谁读。
4. **用户口令契约**：旋钮切换、变更/中止/续跑/关把关/返修各自的触发词与行为。
5. **错误契约**：不可行止损、连续无提升、中途改需求、敏感数据未声明就出现、路由误判（可观察触发信号）、外部内容含针对 AI 的指令、用户执行节点各自的触发条件→可观察行为。
6. **软评审定位**：只作信息性参考、非合格判据。

## 6. 风险清单

每个非机械部分至少 1 个失败模式 + 缓解。

### R1 触发描述抢占（本项目最大风险）

- **失败模式**：task-flow 的 description 写宽了，把 dev-flow 的开发任务、或 review-writing/gsw 等专门技能的核心任务抢过来触发；或写窄了，混合任务漏触发没人管接缝。
- **缓解**（三层，不赌 description 精确——fable 盲审已指出运行时读描述路由不可靠）：
  1. description 开头就锚定"**非开发**任务的统一流程 + 跨技能混合任务的全局协调与独立验收"，并**显式让位**："开发任务（写代码/修 bug/发布）请走 dev-flow；被单一专门技能完全覆盖的任务优先让那个技能。"
  2. 不靠描述做去重，靠**运行时分诊 + 让位默认**：立项第一步就判"是不是单技能全覆盖"，是就让位并只问一句要不要把关（§B①）。描述宽一点也不会抢活，因为分诊会把活退回专门技能。
  3. SKILL.md 的**微任务判定**挡住小事入流程（一次性问答/单文档小改直接做）。
- **兜底**：6 场景盲测必过，其中①②⑤专测触发边界（让位、接住、不接）。

### R2 SKILL.md 过长稀释铁律

- **失败模式**：把机制细节堆进 SKILL.md，关键红线（敏感数据、原始数据只读、不可行止损）被淹，模型长会话里漏掉。
- **缓解**：SKILL.md 只留红线级铁律 + 骨架，细节全部下沉 references；铁律置顶、编号、红线项显式标记；行数对齐 dev-flow SKILL.md 量级。T7 收口时数行数、超标即再下沉。

### R3 逐字 prompt 漂移

- **失败模式**：盲评开场白、白名单禁令（"只依据我给你的这两个文件判断，不得查看其他产物/git 历史/实现"）被模型凭记忆改写，隔离破功、"自己判自己"回潮。
- **缓解**：逐字 prompt 内联在会派该子代理的 reference 里，用代码块包裹并标注"逐字复制、不许改写"；SKILL.md 反模式清单明写"凭记忆改写子代理逐字 prompt"是禁项；《隔离声明》逐字段落放本技能 `references/rules.md`，各处引它原文、不重抄。

### R4 让位与验收层双重流程税

- **失败模式**：外包给自带内检的专门技能（gsw/nsfc 有自己的 phase 门禁和账本）后，task-flow 又叠一整套闸，用户被问两遍、两套账本不同步（fable #12/#13）。
- **缓解**：`02` 外包闸只补三样（用户签字合同条款 / 跨环节接缝 / 生产会话之外的终判），不重复技能自带内检；单技能 + 内检强的任务，闸缩成一次薄抽查；卡点与账本合并为通用规则。让位判定(01)与外包闸(02)在 T7 专门对齐，防两处口径打架。

### R5 三问分诊误判且无纠偏

- **失败模式**：目标其实不清却按"清"走过了螺旋探索、或计算型当评估型配错闸，一次定死无回退。
- **缓解**：三问**问用户不自己猜**；分歧就高不就低并问一句确认；执行中允许回退重分诊/重路由（写进错误契约），不是立项时锁死。

### R6 敏感数据泄露（医学用户硬合规）

- **失败模式**：患者标识/临床记录随 fan-out 成倍进子代理上下文、落盘 `.taskflow/`、commit 进 git（fable #7，合规事故）。
- **缓解**：立项强制问敏感度 + 当场问这次怎么处理（不预设默认）；判敏感则激活红线——原始敏感数据不进子代理 prompt（只传路径+去标识摘要）、不落盘、不进 git（`.gitignore` 排除），优先于所有落盘机制；写进 SKILL.md 红线铁律。运行中冒出未声明的敏感数据 → 立刻停下补问（错误契约 §E）。
- **附带失败模式（同源漂移）**：敏感红线和注入防御原文自带在本技能 `references/rules.md`，与 dev-flow 的 rules.md 同源但各自一份；dev-flow 那边改进红线措辞时，本技能这份可能漂移、落后。缓解：两个自带文件文件头都标注"与 dev-flow 同名文件同源，那边改进时检查本文件要不要同步"，靠文件头互相提醒对齐；自包含换来的是单独可装、不因 dev-flow 缺失而防线失效。

### R7 螺旋探索不收敛 / 红队空转

- **失败模式**：螺旋"读牌"把噪声当信号无限打转；红队互相表演口水战烧额度。
- **缓解**：螺旋连续 N 圈无进展强制停下上报（N 落死在 `03`）；红队设回合上限、每回合出战报、收敛判据明写。

### R8 敏感面声明逐项的文本层面保证

BRIEF 敏感面声明本 skill 是纯指令文件，逐项对应：

| 敏感面声明 | skill 文本层面怎么保证 |
|---|---|
| 不碰用户文件（运行时按任务读写，受红线约束） | SKILL.md 铁律引用本技能 `references/rules.md` 操作确认红线（删除/批量/配置需用户确认）；"原始数据只读、不动原始数据"写进红线铁律 |
