**第 7 条详述(是否有流程指令与红线打架):**

- **BRIEF 点名的"证据账本 vs 不落盘"冲突——设计上不打架**:02 §2 账本登记的是**来源指针**(文献 DOI / 数据文件路径 / 脚本输出 / 在线库查询快照),不是原始数据内容;且铁律 1 明写敏感红线「**优先于所有落盘机制**」,04 §4 再兜底。冲突靠优先级消解,成立。
- **建议级加固(防遗忘)**:三个闸门章节(EVIDENCE §2 / VERIFY §3 / 抽账终判 §6)**没在本节就地重申敏感例外**。三个残余风险点靠"跨文件回看铁律 1"才闭合:
  1. **关键数字本身即敏感**(如患者化验值)登记进 EVIDENCE.md → 触碰不落盘;
  2. **在线库查询快照**含受控记录写进 VERIFY.md → 触碰不落盘;
  3. **抽账终判**(02 §6)把"产出物"整份传给盲评子代理——若产出物含敏感衍生数据、且用户选了"绝不进子代理",就与红线冲突。rules.md §3.1 有兜底(「用户若明确选'绝不进子代理',则该环节改由主会话自己做」),但 02 §6 没提这个回退。
- 三点都由铁律 1 的优先级 + rules.md §3.1 兜住,**不是缺控制,是缺就地提醒**。建议在这三个章节各加一句"敏感数据按铁律 1 走三不,抽账退化为主会话自查",把闭合从"跨文件回看"降为"就地可见"。

## 对照 BRIEF 敏感面声明的核心判据

BRIEF §48-50 声明:纯指令文件、不碰用户文件(运行时按任务读写、受 rules.md 红线约束)、无密钥、不外发;运行时红线=敏感度每次问 / 孤儿清单 / 原始数据只读。

**核对结果:skill 指令实际会让 AI 动的东西未超出声明范围。** 写盘限 `.taskflow/` + git 交付物(属"运行时按任务读写")、原始数据只读、无密钥、不外发,三条运行时红线(01 §4 敏感度强制一问、02 §2 孤儿清单、铁律 2 原始数据只读)都在指令里落实。无"功能正常但越界"的高危项。

## 建议清单(均为建议级,非发布前置)

1. 给常规执行子代理(分析/整理类)补一个带《数据与指令隔离声明》的逐字 prompt 模板,或在 PLAN 派工处加一句"执行子代理 prompt 必带 §0 声明"的显式提醒,别只靠主会话记铁律 6。
2. 在 02 §2 / §3 / §6 三个闸门章节各加一句敏感数据就地提醒:关键数字/查询快照/产出物若含敏感内容,按铁律 1 三不处理,抽账终判退化为主会话自查(呼应 rules.md §3.1 兜底)。
