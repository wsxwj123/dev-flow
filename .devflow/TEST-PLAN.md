# TEST-PLAN — task-flow 验收测试(人话版)

一共 **34 条**测试。每条都是"给装了 task-flow 的 AI 说一句话,看它做得对不对",供裁判盲测(裁判只看行为、不知道谁出的题、也不看实现代码)。详细的判分标准在 `tests/acceptance/test-prompts.md`。

**覆盖了哪些方向**
- 正常该做对的事(9 条):6 个必测场景 + 通用总结报告 + 评估比方案 + 长文先出样品。
- 边界情况(5 条):空话、超大任务、同名任务重复做、三个数据集一起跑、断了接着跑。
- 出错时该怎么办(9 条):任务根本做不了、中途改需求、外部资料里夹着骗 AI 的话、跑着跑着冒出病人隐私、连续几圈没进展、计划步骤排错了、一开始类型判错了、遇到要人做湿实验、评估吵不出结论。
- 断言"不该发生的事没发生"(7 条):小问题被小题大做、开发任务被抢、该交给专门技能的没交、小问答乱写文件、清单锁了又被偷偷改松、把别人技能的内部步骤抄进自己计划、说好不给子代理的敏感数据还是给了。
- 替用户想到的(4 条):拿旧稿+意见回来返修、模型试读的意见别当硬标准、用户说不用把关就别硬挂、拿不准大小就先问一句。

**哪几条是"替用户想到的"(6 场景没提、但按约定必须对)**
- T07 通用总结报告 → 走默认序列,别乱加"重跑验证/红队"这些用不上的环节。
- T08 评估比方案 → 该走红队攻防,有回合上限。
- T31 拿旧稿+导师意见回来 → 直接局部改,别从头重新立项;论文交给 revise-sci。
- T32 模型试读说"不喜欢" → 只当参考,客观检查过了就算合格,不能拿它当门槛打回。
- (另外 T33 用户喊停把关、T34 拿不准就问,也属于常理补位。)

**明确没覆盖什么(不假装全覆盖)**
- **跨平台移植**:OpenCode / Codex 上的文档级兼容,不是"说一句话看行为"能测的,得靠实测环节单独验,这里不设条。
- **纯约定级机制的底层实现**:LOCK hash 到底怎么算、git 是不是真的一个逻辑一个 commit——这些是"君子协定",盲测只能看它行为上有没有守约(T28 测了"锁了不许偷改"这个行为面),不逐位验哈希。
- **真实数据跑通**:所有测试测的是"AI 的决策和动作对不对",不是"分析结果科学上对不对"——后者要真实数据和领域专家,不在本轮验收范围。
