# 04 开发

目标：在 feature 分支上按方案实现，自带白盒单测。

## Step 0：问用户选模型（本阶段第一件事）

问：**代码撰写代理用 opus 还是 fable？**（按你当前额度定；调研/审查/裁判类固定 opus 不用问。）同一项目问过一次后默认沿用，用户随时可改。

## Step 1：开分支

```bash
git checkout -b feature/<功能名>
```

分支已存在（验收打回重修时）→ 直接 `git checkout feature/<功能名>` 切过去，不重复新建；打回循环只从 Step 2 重新派开发代理，不重跑 Step 0/1。

## Step 2：派开发代理

给它的输入：PLAN.md + rules.md 路径 + 本次负责的模块任务。

**Prompt 里必须包含的禁令**："`tests/acceptance/` 目录只读——可以运行里面的测试，严禁修改任何一个字。改了整次开发作废。"

要求：

- 按 PLAN 的任务拆解干活；大任务按模块串行派发（有依赖顺序时），互不依赖的模块可并行派多个代理（各管各的文件，避免冲突）
- 每个模块完成时**自写白盒单测**（`tests/unit/`）并跑通——单测是开发代理的自查工具，覆盖内部逻辑和边界
- 可以自跑验收测试（只读运行），过不过都**如实报告**，不许美化
- 遵守 rules.md：最小改动、一次改一个改完就测、代码风格匹配、危险操作先停下问用户
- 发现 PLAN 有问题（做不了/有更好做法）→ 停下上报主会话，不许自作主张偏离方案

## 完成标准（按项目类型）

- **代码/网页项目**：单测全绿 + 自跑验收测试全绿 + 代码已 commit（一个逻辑变更一个 commit）→ 进入 05。自跑验收有不过的也可以提交验收（05 会给正式判定和打回意见），但要如实说明。
- **skill 项目**：SKILL.md + references 文件齐全、对照 TEST-PLAN 每条测试 prompt 自查"能产出、无明显缺漏"、已 commit → 进入 05（没有单测概念，实测打分由 05 裁判盲测完成）。
- **多平台要求**：本机平台自测通过即可提交验收；其他平台的判定归 06 的 CI 承担（见 05/06）。
