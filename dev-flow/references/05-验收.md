# 05 验收

目标：由独立裁判判定"做完没有"。开发代理说"我做完了"不算数。

## Step 1：核查测试完整性（主会话执行，一票否决）

```bash
git diff $(cat .devflow/LOCK) -- tests/acceptance/
```

输出必须为空。**有任何改动 = 直接判不合格**，打回 04，并向用户报告"开发代理动了验收测试"。

**同时核查"测试运行环境"没被做手脚**（只锁文件内容防不住这招——比如项目根放个 conftest.py 就能悄悄改变验收测试的行为）：检查锁定后对这些文件的改动是否正当——根目录及 tests/ 下的 `conftest.py`、`pytest.ini`、`setup.cfg`、`pyproject.toml` 的测试配置、`package.json` 的 test 脚本。发现针对验收测试的钩子、替身（mock）、路径注入 = 同样判不合格。

## Step 2：干净跑测试（主会话执行）

- 验收测试**单独进程运行**（不和单测混跑一条命令），完整输出落盘到 `.devflow/test-output.txt`；单测另跑一遍一并记录
- 网页项目：起本地服务后跑浏览器自动化；skill 项目：本步只收集材料，实测放 Step 3 裁判做
- 多平台要求的项目：本步只判**本机平台**，跨平台用例标注"待 CI"，由 06 的 GitHub Actions 多平台矩阵判定（本机跑不了别的系统，别硬装能测）
- 如实记录，失败不许重跑到碰巧通过为止（碰运气通过=不稳定=没过）

## Step 3：裁判代理（opus，独立子代理，盲判）

给它的输入：BRIEF.md + TEST-PLAN.md + test-output.txt 路径。**不给**开发过程的任何对话、不给开发代理的自我报告。**Prompt 加禁令**："只依据给你的文件判断，不得查看 git 历史或项目其他文件。"

要求：

- 逐条对照 TEST-PLAN 判定 过/不过
- skill 项目：裁判按测试 prompt 清单**实际调用 skill**，按评分标准给每条产出打分
- 产出 `.devflow/ACCEPT-REPORT.md`：每条判定 + 失败原因 + 总体结论（合格/不合格）

## Step 4：打回循环（自动，不烦用户）

不合格 → 把失败用例和裁判意见回传 04 的开发代理重修 → 修完回到 Step 1。

**循环上限 3 轮**：超过说明方案或测试本身可能有问题，停下向用户报告卡在哪、你的判断是什么。

## Step 5 ⏸ 卡点3：验收报告

全绿后向用户汇报（说人话）：一共测了多少条、几轮修过的、哪些用例修了什么、还有什么已知局限。用户确认 → 进入 06 发布。
