# 07 收尾

目标：用户亲手确认真实效果，同步知识，清理现场。"GitHub 上发布了"不等于做完了。

## Step 0：部署实测（主会话执行，交给用户前先自己跑一遍）

"开发机上代码能跑"≠"装到目标环境能用"。验收/单测测的是核心逻辑，测不出部署环节的坑。交卡点5前，主会话先按**真实部署路径**过一遍，能自查掉的坑别让用户替你踩：

- **照 README 从零跑一遍安装/部署**（真跑 install 脚本或部署命令），不是复用开发时的现成状态。
- **驱动真实集成入口里能自动化的部分**：装完核验——被右键/服务/后台进程集成调用的**底层脚本能独立跑通**；常驻进程/系统扩展在跑的确实是**本次新构建**（`pgrep` 看有没有旧实例赖着用老代码）；产物路径、运行时 PATH/依赖跟开发机一致。
- **常见部署坑清单**（单测/验收覆盖不到，专在这查）：常驻 App/扩展重建后旧实例没杀、安装脚本漏拷文件或路径写死、运行时 PATH/解释器与开发机不同、下载物被 quarantine、签名/权限不足。
- 必须 GUI 手动点的部分（右键菜单、系统弹窗）主会话驱动不了 → 交用户时**列出具体要看的部署失败迹象**，别只甩一句"你试试"。

（真实教训：右键集成"点了没反应"，根因是重装没杀旧 App 进程、跑的还是调已删脚本的老代码——这类坑单测/验收一个都测不出，只有部署实测能逮住。）

## Step 1 ⏸ 卡点5：用户实测（按项目类型给用户可操作的验收方式）

| 类型 | 给用户什么 |
|---|---|
| 网页（已部署） | 线上地址，请用户亲自打开点一遍核心流程 |
| 程序/软件/脚本 | 具体运行命令（从安装到跑起来，照 README），请用户**在自己机器上实际跑一遍**；有右键/服务/GUI 等**真实集成入口的，让用户驱动那个入口，不只跑核心命令** |
| skill | 请用户开新会话真实调用一次，看产出是否符合预期 |

用户说不行 → 把问题记录下来，回 02 的修 bug 模式走一轮。
用户确认没问题 → 继续。

## Step 2：知识同步与复盘

1. 调用 `neat-freak` 技能：同步代码、文档（README/CLAUDE.md）、记忆三端一致
2. 复盘写进项目根 `LEARNINGS.md`：这次踩了什么坑、什么做法值得沿用（按自我进化规则：项目专属写本地，跨项目通用写 `~/.learnings/`）

## Step 3：清理（列清单，经用户确认后执行）

- 删除已合并的 feature 分支（本地 + 远程）
- 残留的 worktree 用 `git worktree remove` 清掉（严禁 `rm -rf`），再 `git worktree list` 确认干净
- 删除临时文件、临时数据
- `.devflow/` **保留**——它是项目档案，下次加功能/修 bug 从这里续接

## Step 4：结束报告

一段人话总结：做了什么、在哪能用、测了什么、遗留什么。流程结束。
