# 跨平台审查清单

**何时用**：BRIEF 声明项目需全平台通用（macOS / Windows / Linux 同时跑）时，05 验收的 `code-review` 抽查逐条对照本清单。专抓"本机全绿、换系统就炸、而 CI 矩阵未必每条路径都触发到"的隐患，这类坑靠人眼审查比靠自动测试更靠得住。

标 ⭐ = 最高频雷区，优先查。每条格式：**症状（哪个平台怎么炸）→ 坏味道（代码长什么样）→ 做法**。

## 路径与文件系统

- ⭐ **路径分隔符 / vs \\**：手拼 `dir + '/' + name` 在 Win 断路径 → 坏味道：字符串 `+` 拼路径、`os.path` 混 `pathlib` → 用 `os.path.join`/`pathlib.Path`/Node `path.join`。
- ⭐ **写死绝对路径**：`/Users/xxx`、`/tmp`、`C:\` 换机即失效 → 坏味道：硬编码用户名/盘符/临时目录 → 家目录 `Path.home()`/`expanduser('~')`，临时目录 `tempfile.gettempdir()`/`os.tmpdir()`，PATH 分隔用 `os.pathsep`（Win 是 `;` 不是 `:`）。
- ⭐ **文件名大小写敏感**：`import Utils` 引 `utils.py`、`require('./Foo')` 在 Mac/Win 照跑、Linux 找不到文件 → 坏味道：import/require/文件引用的大小写与真实文件名不一致 → 统一小写或严格对齐真名，CI 跑一次 Linux。
- **文件名非法字符**：含 `: * ? " < > |` 或结尾空格/点的名字在 Win 无法创建 → 坏味道：用时间戳 `10:30:00`、URL、标题直接当文件名 → 统一 slug 化过滤。
- **Windows 保留名**：`con`/`prn`/`aux`/`nul`/`com1`/`lpt1`（含带扩展名变体）在 Win 不能作文件名 → 坏味道：按用户输入或枚举值直接建文件/目录 → 黑名单校验。
- **路径长度上限**：Win 默认 260 字符截断/报错，深层 `node_modules` 重灾区 → 坏味道：深目录 + 长文件名拼接无长度意识 → 缩短根路径或启用长路径支持。
- **遍历顺序假设**：`os.listdir`/`glob` 返回顺序各平台/文件系统不同 → 坏味道：靠遍历顺序决定处理次序或"取第一个" → 显式 `sorted()`。
- **符号链接/权限位**：`chmod 0o755`、创建 symlink、依赖可执行位在 Win 无效或需管理员 → 坏味道：代码里 chmod/os.symlink/依赖脚本可执行位 → Win 分支跳过或改用启动器 `python x.py`。
- **文件名 Unicode 规范化**：Mac（NFD）与 Linux/Win（NFC）对带音标/中文文件名字节表示不同，跨平台比对失败 → 坏味道：字符串等值比对文件名做去重/查找 → 比对前 `unicodedata.normalize('NFC', ...)`。
- **大小写不敏感文件系统藏 bug**：Mac/Win 默认不区分大小写，`Foo.txt` 与 `foo.txt` 视为同一文件，覆盖/漏读 → 坏味道：靠大小写区分两个文件 → 不用大小写做唯一键。

## 编码与行尾

- ⭐ **默认文本编码**：`open(f)` 读文件 Mac/Linux 走 UTF-8、Win 走 GBK/cp1252，非 ASCII 乱码或 `UnicodeDecodeError` → 坏味道：`open()`/`read_text()` 不带 encoding → 统一显式 `encoding='utf-8'`。
- ⭐ **CRLF/LF**：Win 检出 CRLF，shebang、正则 `\n$`、逐行 diff、hash 校验行为不一致 → 坏味道：`split('\n')` 残留 `\r`、文本/二进制模式混用 → `.gitattributes` 固定 LF、读时 `newline=''`。
- **控制台输出编码**：`print` 中文/emoji 在 Win cmd(cp936) 报 `UnicodeEncodeError` → 坏味道：直接 print 非 ASCII、无 `PYTHONIOENCODING` → 设 UTF-8 输出流或 `sys.stdout.reconfigure`。
- **subprocess 输出编码**：捕获子进程 stdout 用默认编码解码，Win 下中文乱码/报错 → 坏味道：`subprocess` 取 stdout 不指定 encoding → 显式 `encoding='utf-8', errors='replace'`。
- **转义符号差异**：`\` 在 shell/正则/字符串里各平台处理不同，Win 路径 `\` 被当转义符吞掉 → 坏味道：手拼含 `\` 的命令/正则/路径字符串 → 用原始字符串 + 语言原生 API，别手拼。

## 进程与命令

- ⭐ **shell=True 的 shell 差异**：管道、`&&`、通配符、引号在 cmd/PowerShell 与 bash 语义不同 → 坏味道：`subprocess.run(cmd, shell=True)` 带一长串 shell 语法 → 传参数列表 `shell=False`，逻辑交给代码不交给 shell。
- **命令是否存在**：调 `ls`/`grep`/`which`/`open`/`rm` 等 Unix 命令 Win 没有 → 坏味道：subprocess 硬编码 Unix 工具名 → 用语言原生 API（`shutil`/`os`）替代外部命令。
- **可执行文件后缀/查找**：调 `python`/`node`/`./tool` 忽略 Win 的 `.exe`/`.bat`/`.cmd` 与 PATHEXT → 坏味道：硬拼可执行名不带后缀、写死 `/usr/bin/xxx` → `shutil.which()`/`sys.executable` 解析。
- **环境变量语法**：`$VAR`(Unix) vs `%VAR%`(cmd)，且 Win 变量名大小写不敏感 → 坏味道：字符串里嵌 `$HOME`、假设 `Path` 与 `PATH` 不同键 → 用 `os.environ` 读，不手拼语法。
- **信号/进程控制**：`os.kill`+`SIGTERM`/`SIGKILL`、`fork`、`os.setsid` 在 Win 缺失或语义不同 → 坏味道：直接用 POSIX 信号名、依赖 fork → 用 subprocess terminate/kill、避开 fork。

## 依赖与运行时

- ⭐ **平台相关原生依赖**：锁文件钉死某平台预编译包（`*-win_amd64`、`darwin-arm64`），换平台安装失败 → 坏味道：lock 出现单平台 tag、`optionalDependencies` 只列一个平台 → 多平台生成 lock，CI 三系统各装一遍。
- **时间/时区**：`datetime.now()` 无时区、依赖本地时区，不同机器结果漂移 → 坏味道：naive datetime、按本地时间做逻辑 → 统一 UTC + tz-aware。

## 并发与网络

- **文件锁语义**：`fcntl`(Unix) vs `msvcrt`(Win)，Win 独占锁更严（打开中的文件不能删/改名）→ 坏味道：直接用 `fcntl.flock`、假设文件可边读边删 → 用跨平台锁库（portalocker/filelock）。
- **端口/localhost 绑定**：特权端口(<1024) 或 `0.0.0.0` 权限/防火墙行为各异，`localhost` 解析 IPv4/IPv6 不一致 → 坏味道：硬编码低端口、假设 localhost=127.0.0.1 → 用高位端口、显式绑定地址。
- **GUI/字体/DPI**：写死字体名（`PingFang`/`Segoe UI`）、高 DPI 缩放差异 → 坏味道：硬编码系统字体、按平台猜控件尺寸 → 字体回退列表、相对/DPI 无关布局。
