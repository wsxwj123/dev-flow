# task-flow 优化方案 v2（吸收盲审后）

> v1 见 [草案-v1.md]，盲审见 [盲审报告-fable.md]。本文件只记 v1→v2 的**改动**，未提及处沿用 v1。

## 一、两个方向性改动（动骨架，最重要）

### 改动 A：外包不等于撒手——加"独立验收层"（解盲审#1）

**问题**：v1 里路由铁律1（单一技能全覆盖就让位）+铁律2（外包只轻验收）导致最贵的质检（rubric/双轨/裁判/迭代）只在"没有专门技能能干"的边角料任务上启动，而写综述/论文/标书这些高频任务反而躲开了质检。

**改法**：把"让位"和"验收"拆成两件事，重定义三档协作模式——

| 协作模式 | 何时用 | task-flow 做什么 |
|---|---|---|
| **完全让位** | 任务简单且被单一技能全覆盖，用户没提额外质量要求 | 不进流程，直接调那个技能。这是防冲突的根，但适用面收窄——**只在用户没有明确"要严格把关"意图时** |
| **托管执行 + 独立验收（新增，默认档）** | 高频学术任务（写综述/论文/标书等）：由专门技能生产，但用户要一道独立把关 | 专门技能当执行器负责产出；task-flow **在其产出之上叠加独立验收层**：锁定的 rubric 盲评 + 引文/证据抽查 + 读者模拟。生产者和验收者天然分离（专门技能看不到 rubric），正好符合"出方案的≠判定的"内核 |
| **全流程自管** | 无技能覆盖或多技能拼接的任务 | v1 的完整七阶段 |

**关键**：rubric 验收和读者模拟从"内嵌在自管流程里"抽出来，做成**能挂在任何产出物上的独立验收组件**。这样质检价值第一次真正覆盖到高频任务。让位那一档只留给"又简单又不要求把关"的情况。

**谁来判用哪档**：立项拷问时直接问用户"这个任务要不要我独立把一道关"，不靠 task-flow 自己猜。判定权交用户，避开盲审#2 说的"路由误判无纠偏"。

### 改动 B：隐私分级红线（解盲审#7，医学用户硬合规）

新增一条**优先级最高、压过所有落盘机制**的铁律，放进立项第一步：

- 立项强制问一句：**本任务涉及的数据敏感度**（公开 / 一般内部 / 敏感即含患者标识/临床记录/受控数据）。
- 判为敏感 → 自动激活红线，全程有效：
  1. 敏感原始数据**不进任何子代理的 prompt**——只传文件路径和去标识后的摘要/schema，子代理在本地读文件自己处理，不把内容回吐进上下文
  2. 敏感数据和含敏感数据的中间产物**不落盘到 .taskflow/**、**不进 git**（.gitignore 强制加敏感数据目录；交付物里的患者标识去标识后才可入库）
  3. 证据台账登记"数据在哪、谁能访问"，但不登记内容本身
- 这条写进 rules 层，和 dev-flow 的"不删用户文件"同级，违反即流程失效。

## 二、补异常路径（v1 缺的出口，解盲审#5/6/10/12/14）

v1 骨架只有"顺利往下走 + 迭代到合格"，真实科研要补四条岔路：

1. **不可行止损**（#5）：任何执行子任务可返回"做不了 + 证据"（数据不够/样本量不足/资源拿不到），不硬编一个假合格产出，直接触发人工卡点上报用户拍板。
2. **变更单**（#6）：用户中途改需求，不推倒重来。走变更单：记录变更点 → 圈出受影响的子任务和 rubric 条目 → 只解锁受影响部分重跑，已完成且不受影响的保留。
3. **用户执行节点**（#10）：PLAN 支持标记"这步必须用户亲手做"（湿实验、受控数据下载审批、伦理、签字）。到点流程 park，给用户一张可执行清单，用户做完回来续跑，不假装 agent 能做。
4. **返修入口**（#14）：新增一个不从立项起步的入口——输入=已有交付物 + 外部意见（导师 Word 批注、合作者意见），直接进"拆解受影响范围 → 局部迭代"。论文类返修让位给 revise-sci，普通报告/PPT/分析走这个通用返修入口。

外包监控（#12）：托管执行档里，专门技能每完成一个内部阶段回报一次里程碑，工头据此早停，不等最后验收才发现跑偏。

## 三、把四个"想当然"改务实（解盲审#3/4/8/9）

1. **硬验证分轻重**（#3）：
   - 轻量可复现（小数据、秒级）→ 验收代理真重跑，比对结果
   - 重量级流水线（大数据、长耗时、GPU、外部库）→ **不真重跑**，改为核查"确定性证据链"：输入数据哈希、pin 死的包/库版本、完整运行日志、关键中间产物的 checksum
2. **可复现标准放宽**（#4）：从"逐位一致"改为"**容差内一致 + 环境完全锁定**"，并记录是否开了确定性模式。GPU/多线程任务不再因浮点漂移误报不合格。
3. **在线数据源登记快照**（#8）：数据源是活的在线库（PubMed/ChEMBL/ClinicalTrials 等 MCP）时，"原始数据校验值"换成"**查询语句 + 时间戳 + 库版本/release 号**"的快照登记，不强求 checksum。
4. **证据台账分文献/非文献**（#9）：
   - 文献类 → paper-search 验 DOI（沿用 v1），**优先读用户 Zotero 库**取元数据再补检索（#19）
   - 非文献类（临床指南、基因组版本、数据库 release、政府统计、网页）→ 登记 URL + 访问日期 + 版本号，评审代理核对"文里的声明和来源是否一致"，不要求 DOI

## 四、rubric 机制两处修正（解盲审#16/17）

1. **可行性校验**（#17）：rubric 定稿前，除了看 BRIEF，额外过一遍实际数据/资源约束——防止定出"报告 95%CI"这种样本量不够时永远交不出的标准。这一步不破坏黑盒（仍不给 rubric 代理看 PLAN 全文），只喂"数据规模/资源清单"这类约束事实。
2. **写作型 rubric 可经卡点修订**（#16）：写作任务允许迭代中发现更好结构时修订 rubric，但必须经人工卡点 + 留痕（改了哪条、为什么）。计算型 rubric 仍硬锁。区别对待：计算型标准客观、该锁死；写作型标准主观、锁死反而有害。

## 五、成本与并行（解盲审#15/18）

1. **成本预估**（#15）：立项按任务分级预估会起多少个高推理子代理、大致调用规模，明确告诉用户；允许用户按预算砍角色（如跳过反方代理、跳过读者模拟）。把"流程税"从隐性变显性。
2. **多任务隔离 + 残局清理**（#18）：.taskflow/ 按任务 ID 分子目录，PROJECT.md 账本支持多任务条目；任务中止或收尾都走一张残局清理清单（中间大文件、临时 git 分支、park 状态）。

## 六、保留不改（对盲审的反驳）

- **读者模拟**（#20）：不降级为纯参考也不升为硬门，维持"交付前一道非合格判据的信息性提醒"。学术稿的合格判据仍以 reviewer-simulator 的锚定证据批评为准。审查代理其实也是这个意思。
- **样品先行 / 有界迭代 / 交付物走 git**：v1 原样保留，盲审未质疑。

## 七、更新后的骨架

```
入口（三选一）：
  A 新任务   → 01 立项
  B 返修回流 → 直接进 04 前的"局部拆解"（跳立项）
  C 接着上次 → 读 PROJECT.md 续跑

01 立项拷问：交付物形态 / 给谁看 / 什么算好 / 原始材料清点
            + 【新】数据敏感度分级（触发隐私红线）
            + 【新】要不要独立把关（决定走"让位"还是"托管+独立验收"）
            + 【新】成本预估与砍角色
02 拆解与路由：独立拆解代理 + 盲审；每子任务标类型(计算/写作/决策/混合)
              + 标协作模式(让位/托管验收/自管) + 标"用户执行"节点
              ⏸卡点1
03 验收标准先行：rubric 可判定强制 + 可行性校验 + 锁定
              （写作型可后续经卡点修订，计算型硬锁）⏸卡点2
03.5 样品确认（大交付物才有）
04 执行：按协作模式派工；托管档带里程碑回报；
        任何子任务可返回"不可行+证据"止损；用户执行节点 park
05 双轨质检：硬验证(分轻重) + 软评审(锁定rubric盲评) + 裁判 + 有界迭代
           ⏸卡点3
        中途改需求 → 走变更单，局部解锁重跑
06 读者模拟(非硬门) + 交付清单 ⏸卡点4
07 收尾复盘 + 残局清理 + LEARNINGS
```

产物 .taskflow/<任务ID>/：BRIEF / PLAN(含路由表+协作模式+节点标记) / RUBRIC+LOCK / EVIDENCE(分文献/非文献) / VERIFY / REVIEW-r{n} / CHANGE(变更单) / ACCEPT-REPORT / DELIVERY。敏感数据目录 .gitignore 强制排除。

## 八、已拍板的落地决策（2026-07-16）

| # | 问题 | 决定 | 对设计的影响 |
|---|---|---|---|
| 1 | 独立验收层默认档 | **默认开，可关** | 学术任务默认多派一个独立检查员按锁定 rubric 挑毛病；每次立项/交付前提示，用户可一句话关掉 |
| 2 | 隐私红线严格度 | **不设死默认，每个敏感任务立项时问用户这次怎么处理** | 立项拷问必含"数据敏感度"一问；判为敏感则**当场问用户**这次是"脱敏后可进子代理"还是"绝不进"，按当次回答激活对应红线，不预设 |
| 3 | 样品先行门槛 | **仅长文/大分析强制**，短交付物直接做 | 03.5 样品确认只在大交付物触发；门槛具体阈值留 grilling 细化 |
| 4 | hook 提醒 | **先不定，技能做完再说** | 本轮不做 hook；skill 自身触发词要写扎实，作为唯一触发通道 |

### 仍待正式 grilling 细化
- 技能名（task-flow？还是中文名）
- 样品先行的"长文/大分析"具体量化阈值
- 隐私"每次问"的问法要不要给快捷预设（如记住"这个项目一律脱敏后可进"）
