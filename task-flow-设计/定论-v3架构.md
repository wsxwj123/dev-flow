# task-flow v3 架构定论（吸收 opus 头脑风暴 + 用户参与度定论）

> 前情:v1(工头 C)→ fable 盲审 → v2(补漏)→ opus 头脑风暴(5 个替代范式,指出 C 是"拿 dev-flow 锤子看钉子",建议分诊+可插拔)→ 本文件定架构。

## 一句话架构

**一个轻分诊入口 + 一套极简主循环 + 两道客观质量闸 + 一个随时可调的"参与度旋钮" + 几个按任务类型触发的可选动作。**

不是七阶段重流程一统天下(那是 C 的错),也不是五套独立流程插件(那更重)。

## 三根支柱

### 1. 参与度旋钮（贯穿全程,不是分诊分支）

用户定论:"想参与就参与,不想就不参与"——参与度是可变的,不能立项定死。

- **默认放手**:AI 自主拆解→执行→过质量闸,只在关键节点停下给用户看
- **随时一句话切换**:用户说"这步我看看/我来定"→ 立刻切成结对模式(做一小步就摊开+等用户拨方向,即 opus 的 E 范式内核);说"继续放手"→ 切回自主
- 切换不重新立项。这是 E 范式的正确用法:不做成独立流程,做成随时可入可出的档位

### 2. 轻分诊（立项时三问,决定主循环怎么配)

参与度拿掉后,分诊只剩三个客观维度:

| 问 | 答案影响 |
|---|---|
| 目标清不清楚? | 不清 → 主循环前置"螺旋探索"几圈(B 范式内核:押注+最小验证,把目标迭代出来),清了再转正式产出 |
| 有没有客观对错? | 有(计算型)→ 质检重心在硬验证闸;无(评估型)→ 加一轮对抗红队(D 范式内核) |
| 错误代价高不高? | 高 → 质量闸全开、样品先行强制;低 → 简配 |

### 3. 两道客观质量闸（替换 C 的"一堆评审代理判质量"）

opus 最有价值的点:C 最该换的不是骨架,是质检那一环的**赌注**。别赌"多派评审代理能判好坏"(模型判模型,天花板低),改赌客观机制:

- **契约闸门(A 范式)**:立项写死一张"出厂检查单",每条是可机械执行的检查动作(引文能否逐条查到 DOI、数字和原始数据对不对得上、有没有超出数据的结论、格式合规)。产物不过闸不进交付。硬闸(造假/动原始数据/数字对不上)零容忍,软闸给扣分位置让用户定
- **证据账本(C' 范式)**:每句实质论断拴证据,拴不住的句子进"孤儿清单"现形;交付前抽账核对(来源真存在 且 真能推出这句话)。质量从可追溯性涌现,不靠主观评审

软评审(模型盲评)降级:从"合格判据"变成"信息性提醒",只在两道客观闸之外补软质量(洞见/文采)的参考意见,不作数。

## 主循环（极简,替代七阶段）

```
立项(三问分诊 + 写出厂检查单 + 数据敏感度问一次) ⏸
  │
  ├─目标不清? → 螺旋探索几圈,收敛后继续
  │
拆一步 → 做一步(自主 或 结对,看旋钮) → 过质量闸(契约+账本)
  │         ↑                                    │
  │         └────── 没过,打回改 ────────────────┘
  │
  ├─评估型? → 加一轮红队对抗
  │
  ├─大交付物? → 先出样品确认口味
  │
交付前:抽账核对 + 软评审提醒(非硬门) ⏸
  │
收尾复盘 + 残局清理
```

保留自 v2 的硬约束(红线,不是阶段):
- 不动原始数据;引文/论断必须可溯源;敏感数据处理每次立项问用户(不预设);全程说人话
- 外包给专门技能的产物,同样要过质量闸,但闸门是**查缺补漏式**(用户指正:gsw 等技能内部本就派独立子代理审查,不能当它们没有):立项时先盘点该技能自带哪些检查,闸门只补三样——①用户事先签字锁定的合同条款(技能自带标准是技能定的,合同是用户定的)②跨环节接缝(如论文数字 vs 分析管线输出的一致性,两边技能都不管)③最终抽账+由生产会话之外做"算不算完"的判定。单技能+内检强的任务,闸门缩成一次薄抽查,不叠双份流程税
- 有界迭代:连续两轮无提升强制停下上报

## 相比 C(v2) 的净变化

| | C / v2 | v3 |
|---|---|---|
| 骨架 | 固定七阶段 | 轻分诊 + 极简主循环,阶段按任务裁剪 |
| 质检赌注 | 一堆隔离评审代理盲评打分 | 契约闸门 + 证据账本(客观),软评审降为参考 |
| 参与度 | 5 个固定卡点 | 随时可调旋钮(默认放手,一句话切结对) |
| 探索型任务 | 硬套线性流程(软肋⑤) | 螺旋探索前置 |
| 评估型任务 | 一次性软评审 | 红队对抗 |
| 成本 | 重,小任务不划算 | 按分诊裁剪,轻任务几乎零编排 |
