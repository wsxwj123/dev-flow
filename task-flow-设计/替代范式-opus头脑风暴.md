# task-flow 替代架构范式（opus 头脑风暴，跳出"工头/调度"框架）

派 opus 代理,明确要求它不改进 C、想根本不同的骨架。产出 5 个替代范式 + 总评。

## 5 个替代范式速览

| 代号 | 核心组织原则 | 专治 C 的软肋 | 自身软肋 | 最适合 |
|---|---|---|---|---|
| **A 契约闸门** | 不管过程只卡节点:产物想进交付必须过一张"出厂检查单",谁做的、怎么做的无所谓 | ③流程重 ④路由误判(根本不路由) | 质量全赌检查单;事后拦截浪费大;软质量(洞见/文采)卡不住 | 结果把关、给别人产出做终检、数据分析结果验收 |
| **B 螺旋发现** | 不假设一开始知道目标;循环单位=一个押注+一次最小验证,目标是迭代出来的 | ⑤线性不适合探索 | 收敛不了(N靠拍脑袋);"读牌"要求高、易把噪声当信号 | 探索性数据分析、开放选题预研、方法可行性 |
| **C' 证据账本** | 每句实质论断都拴证据,拴不住不许进交付;质量从可追溯性涌现,不靠评审代理判 | ②软评审天花板(换成客观对账) ①外包也要交账 | 只保下限(每句有据)不保上限(论证漂亮);记账打断节奏 | 综述/论文事实主体、系统评价、临床证据整理 |
| **D 对抗博弈** | 生成方 vs 红队反复交手,打到红队攻不动才收敛;质量来自"扛住持续攻击" | ②(多轮压测 > 一次性打分) | 贵;易空转(互相表演口水战);收敛判据仍是软判断 | 选题/方案评估、可行性论证、投稿前自我攻防 |
| **E 结对副驾** | 换人机分工:人一直在环里当品味裁判和方向盘,agent 只往前挪一小步就摊开给人看 | ①②(人来判,不存在撒手和模型自评) ③(无子代理开销) | 极耗用户注意力;质量上限=用户判断力;大任务不适合 | 中小、探索性、用户是内行且愿参与的任务 |

## 总评核心结论

1. **C 不是唯一最优,是"手上有 dev-flow 锤子所以把通用任务也看成钉子"**。C 的重编排/强流程/多子代理,是软件开发场景(有客观测试、明确验收、边界清楚)逼出来的,这三个前提在很多科研任务上不成立。

2. **按任务分诊,不是竞争关系**:
   - 探索型(不知道要什么)→ B 螺旋
   - 事实写作(真实性>文采)→ C' 账本
   - 评估型(无客观对错)→ D 对抗
   - 中小+内行+愿参与 → E 副驾
   - 结果把关 → A 闸门
   - **只有**多环节+跨技能拼接+错误代价高+验收能提前写清的大型混合任务(如"生信分析+写成投稿论文"端到端)→ C 才真正最优

3. **最优架构很可能是:一个极轻的分诊入口 + 几套可插拔骨架**。立项先判任务属于哪类(目标清不清、有没有客观测试、代价高不高、用户想不想参与),挂最合适的骨架。C 从"唯一流程"降格成"分诊里最重的一档"。

4. **A 和 C' 不是另一条流程,是能挂在任何产出物上的质量层**,可以直接嵌进现在的 C,把 C 软肋②"靠一堆评审代理判质量"替换成"靠对账和过闸"这种客观机制。C 最该换的不是骨架,是质检那一环的**赌注**:别赌"多派评审代理能判好坏",改赌"可追溯性 + 硬闸门"。

## opus 留的反向拷问(值得正面回答)
