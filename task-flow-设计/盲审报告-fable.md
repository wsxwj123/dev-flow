# task-flow 设计草案 fable 盲审报告（原文存档）

审查者：fable 独立子代理（盲审，只知"隔壁同事设计"）
被审对象：草案-v1.md
日期：2026-07-16

## 总结论

方向站得住（工头/调度定位、双轨质检、样品先行、有界迭代都对），但有一个方向性隐患 + 一批会让流程失效的漏洞。最致命：这套质检机制恰好被自己的路由铁律绕过了最常见的真实任务；加上"硬验证一键重跑""运行时读技能列表路由""患者数据零处理"三个想当然，在真实科研场景会成片崩掉。

---

## 第一梯队：会让流程失效或产出错误结果

**1【方向性】质检机器只对少数派任务生效，正好躲开高频任务。**
路由铁律1（单一技能全覆盖→让位）+铁律2（整段外包只做轻验收）合起来 = 写综述/论文/标书这些高频任务要么不进 task-flow、要么只走轻验收。rubric 锁定、双轨、裁判、有界迭代这套重机器只在"无技能覆盖→通用子代理"的残余任务才完整生效。最贵的质检投在最少见的情况上。
→ 明确 task-flow 对"已让位/已外包"任务的质检价值：要么承认是薄壳，要么把 rubric/读者模拟做成可叠加在专门技能产出之上的独立验收层。

**2【B】运行时读技能列表路由不可靠，路由错了全程无纠偏。**
100+ 技能、描述是营销文案（gsw 和 review-writing 都自称"全流程"），LLM 读描述匹配误判是常态。误判让位→整套治理消失，等交付才暴露。
→ 路由判定产出"为什么选它/为什么排除相邻技能"的理由，允许执行中回退重路由，不是只在卡点1否决一次。

**3【B】硬验证"一键重跑=验收"在真实生信/ML 跑不起来。**
比对基因组几小时到几天、TB 级输入、依赖会变版本的外部库（Ensembl/dbSNP release）、要 GPU。验收代理不可能一次会话重跑完。"可复现"退化成无法执行的口号。
→ 区分轻量可即时重跑（小数据/秒级）与重量级流水线（核查确定性证据：固定输入哈希、pin 版本、日志、中间产物 checksum，不真重跑）。

**4【B】固定随机种子≠可复现——GPU/多线程本身不确定。**
GPU 浮点归约、多线程 BLAS、cuDNN 非确定算子，种子固定结果也逐位漂移。硬验证"数字前后一致"会误报。
→ 可复现放宽为"容差内一致 + 环境完全锁定"，记录是否启用确定性模式。

**5【A】完全没有"任务根本做不了"的中止/上报路径。**
数据质量太差、样本量不够、公开数据不存在、受控数据没批。当前只有"迭代到合格"，子代理会硬编看似合格的产出。
→ 执行子任务允许返回"不可行+证据"，触发强制人工卡点。

**6【A】用户中途改需求撞死锁定的 rubric 和 PLAN。**
"审稿人要加亚组分析""导师说换数据集"是日常。BRIEF/PLAN/RUBRIC 全 LOCK，没写怎么解锁、已完成子任务怎么保留、rubric 怎么重发。
→ 加"变更单"：记录变更点、圈受影响子任务与 rubric 条目、局部解锁重跑，不全量作废。

**7【C 隐私，医学致命】患者/敏感数据能不能进子代理 prompt、落盘、进 git，完全没提。**
面向医学用户，任务含 PHI。fan-out 给多子代理=数据成倍进上下文，产物落盘 .taskflow/，"文稿脚本图全进 git"=把患者数据 commit 进 git 是合规事故，多代理传递是二次泄露面。
→ 立项强制数据敏感度分级；敏感数据设"不出本地/不进子代理 prompt/不进 git，只传路径与去标识摘要"红线，优先于所有落盘机制。

## 第二梯队：真实场景常见、明显摩擦

**8【B】原始数据"只读+登记校验值"假设静态本地文件，但数据源是活的在线库。**
PubMed/ChEMBL/ClinicalTrials/bioRxiv 等 MCP 运行时查询、随时间变化，没法 checksum。
→ 在线数据源用"查询快照+时间戳+库版本"登记。

**9【B】引文验真只覆盖文献，非文献事实无验证轨。**
NCCN/WHO 指南、GRCh38 版本、UniProt 条目、政府统计无 DOI，恰是最易写错版本/年份处。
→ 证据台账区分"文献（DOI 验真）"与"非文献（URL+访问日期+版本号，评审代理核对声明与来源是否一致）"。

**10【C 人机分工】没有"必须用户亲手做"的步骤节点。**
湿实验、受控数据下载（dbGaP/UK Biobank 需机构审批）、伦理审批、纸质签字，agent 干不了。PLAN 全按外包/派工建模会卡死或假装完成。
→ PLAN 支持"用户执行"节点，到点 park 给可执行清单，完成后续跑。

**11【C 长周期】跑几天的流水线无脱机/重连。**
子代理派工假设会话内跑完。
→ 重量级子任务改"生成可提交作业脚本+交用户/调度器跑+回收核查"，执行与会话解耦。

**12【A】外包的专门技能干到一半卡住，工头无中途监控。**
gsw 13+ phase 自带卡点，中途报错/跑偏/停在自己卡点，工头等最后验收才发现，前面白跑。
→ 外包设心跳/里程碑回报，每完成内部阶段回报一次。

**13【A/B】专门技能自带卡点+账本与 task-flow 的卡点+PROJECT.md 打架。**
gsw/nsfc 有自己 phase 门禁和落盘状态，外包时两套卡点两个账本，用户被问两遍、状态不同步。
→ "卡点与账本合并"从 dev-flow 特例升级为对所有有内部流程专门技能的通用规则。

**14【C 返修回流】交付物给第三方改回来，没有再入口。**
导师 Word 批注、合作者意见是日常。task-flow 每次从立项起步，没有"接住带批注既有交付物→局部返修"入口（论文有 revise-sci，普通报告/PPT/分析没有）。
→ 加"返修入口"：输入=既有交付物+外部意见，跳过立项直接进拆解受影响范围+局部迭代。

**15【C 成本】标准任务一堆高推理子代理，成本不可见、无预算。**
拆解/盲审/rubric/评审/裁判/反方/读者模拟全高推理，一个标准任务十几次高成本调用，用户会因烧额度弃用。
→ 立项按分级预估调用规模并展示，允许按预算砍角色（跳过反方/读者模拟）。

## 第三梯队：影响质量非致命

**16【B】rubric 锁定对写作有害，锁定后无修订路径。**
更好结构写出来才发现；rubric 只看 BRIEF 可能锁进逼凑字数打勾的表层标准。
→ 写作型 rubric 允许迭代中经卡点修订、留痕，保留防跑偏不锁死结构。

**17【B】rubric 与可行性脱钩。**
rubric 只看 BRIEF 不看数据/PLAN，可能定出"报告效应量 95%CI"这种样本量不足时交不出的标准，永远不合格。
→ rubric 定稿前过一次可行性校验，拿实际数据/资源约束核对每条能否交付。

**18【C 残局+并行】失败任务残局无清理，多任务并行撞账本。**
中止后 .taskflow/、中间大文件、git 分支留一地；多分析并行时单一 PROJECT.md+单写者会命名冲突。
→ .taskflow/ 按任务 ID 分目录，收尾/中止走残局清理清单；账本支持多任务条目。

**19【C 未利用】用户有 Zotero/pyzotero，证据台账应对接而非重新检索。**
→ 证据台账优先读 Zotero，缺失再走 paper-search。

**20【B】读者模拟终审会高估模型的浅层第一印象。**
模型扮读者给第一印象仍是模型判模型，作硬门会漏真问题或凭空挑刺。
→ 读者模拟定位"信息性反馈"而非合格门，学术稿仍以 reviewer-simulator 锚定证据批评为准。

---

## 一句话总纲
方向对，补两个方向性缺口：(a) 把质检价值明确覆盖到"让位/外包"的高频任务，别让重机器空转在残余任务；(b) 补齐异常路径（不可行止损、改需求、外包卡壳、返修回流）和医学隐私红线。再把"硬验证=一键重跑""种子=可复现""数据是静态本地文件""运行时读描述能准确路由"四个想当然按真实场景改务实。
